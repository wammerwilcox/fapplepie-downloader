# 2.0.0 Scraper Hardening Status

## Core Goal

Make scraper failures fast to diagnose and safer to operate while keeping the current lightweight HTTP and `curl_cffi` approach. This release does not add Playwright, Selenium, browser automation, CAPTCHA solving, or broad CLI redesign.

## Scope

Included:

- `--probe` mode for diagnosing the scrape path without downloads or URL/cache mutation.
- Phase-aware diagnostics for base URL probing, robots handling, first-page fetch/parse, pagination, redirect resolution, and download handoff.
- Polite timing controls with scheduled-run start jitter and page/redirect delays.
- Transport isolation around scrape requests so future `curl_cffi` or proxy changes stay local to scrape transport code.
- Docs and tests for the diagnostic and timing behavior.

Excluded:

- Browser automation.
- CAPTCHA or challenge solving.
- Cookie/session persistence beyond existing scraper headers/session behavior.
- Major CLI redesign beyond targeted flags and configuration.

## Remaining Hardening

- Make the scrape page limit configurable. The current scrape loop still uses a hard-coded `max_pages = 100` safety limit.
- Improve download-side diagnostics and retry behavior:
  - Add bounded retry/backoff for likely transient `yt-dlp` failures.
  - Avoid retrying known permanent extractor failures such as "No video formats found".
  - Keep failed downloads out of `downloaded_urls`.
  - Keep successful downloads cached as they are today.
- Document any new downloader environment controls after implementation.
